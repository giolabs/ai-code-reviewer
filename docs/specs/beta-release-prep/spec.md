# Beta Release Prep

> **Status:** DRAFT

## 1. Goal

Prepare the `ai-code-reviewer` npm package for its first public beta (`0.1.0-beta.1`) by fixing five gaps that block or degrade the publish experience: bump the semver to beta, make the `package.json` metadata provider-agnostic, add the missing `LICENSE` file, create the `examples/` folder referenced in the README, and include `examples/` in the npm package bundle.

## 2. Scope

### Included in this phase

- Bump `version` in `package.json` from `0.1.0` → `0.1.0-beta.1`
- Rewrite `description` in `package.json` to be provider-agnostic: `"AI-powered code reviewer for GitHub PRs — supports OpenAI, Anthropic, Gemini and Ollama"`
- Extend `keywords` in `package.json` to add `anthropic`, `gemini`, `ollama`, `multi-provider`; remove `gpt` (implementation detail, not a user search term)
- Add `"examples"` to the `files` array in `package.json` so the folder ships with the package

- Rewrite `examples/code-review-rules.md` — file exists but is in Spanish and only 28 lines; needs a full rewrite: English, rich TypeScript/NestJS content, ~100 lines

- Bumping the version to `0.1.0` final or `1.0.0` — this is beta only
- Any changes to `src/`, `templates/`, `__test__/`, or `dist/`
- Adding new npm scripts
- Creating additional example files (e.g., per-stack variants)

#### `examples/code-review-rules.md`

A rich, opinionated example for a TypeScript/NestJS project. Users copy this file to their repo as `code-review-rules.md` and edit it. The file must:

- Open with a one-paragraph intro explaining what the file is and how to reference it from `.ai-review.yml`
- Cover at least 4 rule sections: **Architecture**, **Security**, **TypeScript conventions**, **Testing**
- Each section has 3–6 concrete, checkable rules — not vague suggestions
- Rules are written in imperative form: "Never import...", "Every controller must...", "Always use..."
- Include at least 2 examples of what the reviewer should flag (with `❌ Bad` / `✅ Good` code snippets)
- Include a brief section on **exceptions** (what NOT to flag in this project)
- The file must NOT be generic — it must read as a real project's conventions, e.g., referencing NestJS decorators, TypeScript strict mode, specific folder structure
- Maximum length: ~100 lines — dense but scannable

Reference style from `templates/nestjs-rules.md` for the rule format and `templates/typescript-rules.md` for TypeScript-specific checks. Do not copy content from those files verbatim — `examples/` is a user-facing example, not a built-in template.

## 10. Decisions Made (Locked)

| Decision | Why |
|---|---|
| Version `0.1.0-beta.1`, not `0.1.0` | Signals pre-release status to npm users; allows iteration before the stable release |
| Remove `gpt` from keywords | A model name, not a user search term; clutters the keyword list |
| Keep `openai` in keywords | OpenAI is still the default provider; removing it would hurt discoverability |
| Copyright holder: `Giolabs`, year: `2026` | Matches `package.json` `author` field; year is the project's first commit year |
| `examples/` uses NestJS/TypeScript project | Most representative of the tool's primary audience; easier to make concrete than a generic example |
| `examples/` added to `package.json` `files` | README links to it; npm users must be able to access it after `npm install` |

## 11. Edge Cases

### Invalid input

Not applicable — no user input at runtime.

### npm publish edge cases

- **`files` field missing `"examples"`**: `npm pack --dry-run` would exclude the folder. Verification step catches this.
- **`LICENSE` missing from `files`**: Already present in the current `files` array — do not remove it.
- **`npm pack` includes unexpected files**: `node_modules/`, `src/`, `__test__/`, `.env`, `dist/` (already excluded by the current `files` whitelist). No change needed.

### Content edge cases

- **`examples/code-review-rules.md` empty or stub**: README link would be useless. The file must have real, opinionated content per section 6.
- **MIT license text incorrect**: npm shows the `license` field from `package.json` (`"MIT"`) but the actual file must match. Use the exact canonical MIT text from section 6.
