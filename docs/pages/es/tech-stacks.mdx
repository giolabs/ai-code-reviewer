# Stacks tecnológicos

El revisor detecta automáticamente el stack tecnológico de tu proyecto a partir de `package.json`, `pubspec.yaml`, `composer.json` y otros archivos indicadores. Luego carga un template de reglas built-in optimizado para ese stack.

## Orden de auto-detección

La detección va de más específico a más genérico, de modo que, por ejemplo, una app de Next.js obtiene las reglas de Next.js y no solo las de React.

| Orden | Stack | Detectado por |
|---|---|---|
| 1 | NestJS | `@nestjs/core` en dependencias |
| 2 | Next.js | `next` en dependencias |
| 3 | React | `react` en dependencias |
| 4 | Flutter | `pubspec.yaml` presente |
| 5 | Laravel | `composer.json` con `laravel/framework` |
| 6 | Node.js | `package.json` presente (sin match de framework) |
| 7 | TypeScript | `tsconfig.json` presente |
| 8 | Genérico | Fallback |

## Forzar un stack

Sobreescribí la auto-detección con `tech:` en `.ai-review.yml`:

```yaml
tech: nestjs
```

Valores válidos: `nestjs` | `nextjs` | `react` | `typescript` | `node` | `flutter` | `laravel` | `generic`

## Reglas por stack

### NestJS (`nestjs`)

- Inyección de dependencias: servicios, repositorios y providers deben inyectarse vía constructor — sin instanciación directa dentro de métodos
- Decoradores: uso correcto de `@UseGuards`, `@UseInterceptors`, `@UsePipes` en controllers y métodos
- Límites de módulo: imports, exports y providers declarados correctamente por módulo
- DTOs: validación de input con `class-validator`; sin acceso directo al request body
- Manejo de excepciones: `HttpException` y filtros de excepción built-in; sin `throw new Error()` directos
- Base de datos: patrón repository de TypeORM; evitar llamadas directas a `EntityManager`
- Seguridad: guards en rutas autenticadas; sin `@Public()` en endpoints sensibles

### Next.js (`nextjs`)

- Modelo de componentes: uso correcto de Server Components vs Client Components; `use client` solo cuando sea necesario
- Data fetching: `fetch` con caché apropiado (`cache: 'no-store'` para dinámico, `revalidate` para ISR)
- Routing: convenciones del App Router (archivos layout, page, loading, error); sin mezclar patrones del Pages Router
- Optimización de imágenes: `next/image` en lugar de etiquetas `<img>`
- Navegación: `next/link` en lugar de `<a>` para rutas internas
- Entorno: prefijo `NEXT_PUBLIC_` para variables expuestas al cliente; sin secrets en bundles del cliente

### React (`react`)

- Hooks: reglas de hooks (sin llamadas condicionales o anidadas); arrays de dependencias completos y correctos
- Estado: estado mínimo; valores derivados computados, no almacenados
- Performance: `useCallback`/`useMemo` cuando realmente sea necesario; evitar memoización prematura
- Prop drilling: context o librería de estado para estado compartido profundamente
- Efectos secundarios: cleanup en `useEffect`; sin async directamente en `useEffect`
- Keys: keys estables y únicas en listas; sin index de array como key en listas mutables

### TypeScript (`typescript`)

- Tipos: sin `any`; tipos de retorno explícitos en funciones exportadas
- Null safety: manejo explícito de `undefined`/`null`; uso de optional chaining y nullish coalescing
- Generics: generics con restricciones; sin `<T>` sin restricción
- Enums: `const enum` para conjuntos cerrados de valores; union types para discriminantes abiertos
- Assertions: sin casts `as Type` excepto en fixtures de tests o type guards

### Flutter (`flutter`)

- Gestión de estado: patrón BLoC/Cubit; sin `setState` directo en widgets grandes
- Separación de widgets: widgets solo de UI; sin lógica de negocio en métodos `build()`
- Async: uso de `FutureBuilder`/`StreamBuilder`; `async/await` en controllers
- Navegación: rutas nombradas o GoRouter; sin cadenas directas de `MaterialPageRoute`
- Platform: guards `Platform.isAndroid`/`Platform.isIOS` para código específico de plataforma
- Performance: constructores `const`; `ListView.builder` para listas largas

### Laravel (`laravel`)

- ORM: relaciones Eloquent correctamente definidas; `with()` para eager loading y evitar N+1
- Validación: clases `FormRequest`; sin validación inline en controllers
- Auth: acceso via `auth()->user()`; autorización basada en policies; sin user ID crudo del request
- Migraciones: métodos `down()` reversibles; sin cambios de schema en seeders
- Rutas: resource controllers RESTful; route model binding
- Seguridad: CSRF en POST/PUT/DELETE; sin SQL crudo sin parámetros vinculados

### Node.js (`node`)

- Async: `async/await` sobre callbacks; propagación correcta de errores (`try/catch`)
- Event loop: sin I/O de archivo/red síncronas en hot paths
- Entorno: acceso a `process.env` detrás de un módulo de config
- Manejo de errores: rechazos de promesas sin manejar; `process.on('unhandledRejection')`
- Seguridad: `helmet` para cabeceras HTTP; sanitización de input

### Genérico (`generic`)

Aplica a cualquier stack. Cubre:
- Seguridad: inyección, XSS, credenciales hardcodeadas
- Performance: complejidad algorítmica, loops innecesarios
- Mantenibilidad: complejidad, naming, duplicación
- Riesgo de bugs: null safety, off-by-one, condiciones de carrera
- Arquitectura: separación de concerns, dirección de dependencias

## Combinando con reglas personalizadas

Las reglas del template built-in siempre se cargan primero. Tu `code-review-rules.md` (o el archivo en `rules:` de la config) se agrega después, así tus reglas ganan en conflictos.

```yaml
# .ai-review.yml
tech: nestjs
rules: ./reglas-equipo.md
```

Ver [Reglas personalizadas](./custom-rules) para ejemplos.
