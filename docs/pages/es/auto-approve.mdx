# Auto-aprobar

Por defecto, el revisor nunca publica un evento GitHub `APPROVE` — solo `COMMENT` o `REQUEST_CHANGES`. Esto mantiene la aprobación humana como una puerta deliberada.

La feature de auto-approve es un opt-in que permite al bot publicar un `APPROVE` real cuando está seguro de que el PR está limpio.

## Condiciones

Las cuatro condiciones deben cumplirse simultáneamente:

1. `autoApprove.enabled: true` en la config
2. La recomendación del modelo es `approve`
3. No hay hallazgos `critical` o `major` en la revisión
4. `overallScore` ≥ `autoApprove.minScore` (por defecto: 7), **o** el modelo no devolvió un score

Si alguna condición falla, el bot vuelve a `COMMENT` y explica por qué no aprobó.

## Configuración

```yaml
# .ai-review.yml
autoApprove:
  enabled: true
  minScore: 7    # escala 0–10; 7 es el valor por defecto
```

No hace falta cambiar el workflow. El comando `review-pr` lee la config y se encarga del resto.

## Descarte de revisiones anteriores

Si el bot previamente publicó una revisión `REQUEST_CHANGES` en el mismo PR, y la nueva revisión califica para aprobación, el bot automáticamente **descarta su propia revisión anterior** antes de publicar `APPROVE`. Esto evita que el PR quede bloqueado por una revisión obsoleta.

Esto usa la API de dismiss-review de GitHub y solo descarta revisiones publicadas por la misma cuenta del bot (`GITHUB_TOKEN`). Nunca descarta revisiones de humanos.

## Umbral de score

El campo `minScore` es un piso de seguridad. Incluso si el modelo recomienda `approve` y no hay hallazgos bloqueantes, un score general bajo (ej. un PR que "funciona técnicamente pero es un desorden") no va a ser auto-aprobado.

Configurá `minScore: 0` para aprobar con cualquier recomendación `approve` independientemente del score. Configurá un valor más alto (ej. `8` o `9`) para equipos más estrictos.

Si el modelo no devuelve el campo `overallScore` (versiones antiguas de modelos o proveedores con output estructurado limitado), la condición de score se omite y las otras tres condiciones siguen aplicando.

## Razones de omisión

Cuando el bot vuelve a `COMMENT` en lugar de `APPROVE`, incluye la razón en el comentario resumen:

| Razón de omisión | Mensaje |
|---|---|
| Feature deshabilitada | *(feature inactiva — sin mensaje)* |
| El modelo no recomendó approve | "Recomendación del modelo: `request_changes` / `comment`" |
| Hay hallazgos critical/major | "Se encontraron N hallazgo(s) bloqueantes" |
| Score por debajo del umbral | "Score X está por debajo del mínimo Y" |

## Combinando con branch protection

Una configuración recomendada para equipos que quieren revisión con IA como gate pero igual requieren aprobación humana:

1. Habilitar auto-approve en `.ai-review.yml`
2. Agregar una regla de branch protection que requiera:
   - Status check: `AI Code Review / ai-review` debe pasar
   - Revisiones requeridas: al menos 1 aprobación humana

Con esta configuración, el bot aprueba automáticamente los PRs limpios, reduciendo la carga cognitiva del revisor humano para cambios sencillos. El revisor humano puede enfocarse en los casos `COMMENT` y `REQUEST_CHANGES`.
