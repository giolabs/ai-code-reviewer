# Feedback inline

La feature de feedback inline permite que los desarrolladores respondan directamente a los comentarios del bot en un PR para obtener más contexto o desestimar hallazgos que ya resolvieron.

## Comandos

| Respuesta | Qué pasa |
|---|---|
| `/explain` | El bot responde con una explicación detallada del hallazgo: por qué importa, qué podría salir mal, y un ejemplo concreto de cómo arreglarlo |
| `/dismiss` | El bot marca el hilo del comentario como resuelto y confirma el descarte |

Las respuestas deben ser en un **comentario de revisión del PR** (comentario inline en el diff), no en un comentario regular del PR.

## Configuración

### 1. Habilitar en la config

Agregá a `.ai-review.yml`:

```yaml
feedback:
  enabled: true
  allowDismiss: true
```

### 2. Agregar el job al workflow

El comando `handle-feedback` debe ejecutarse en el evento `pull_request_review_comment`. Agregalo junto a tu job existente de `ai-review`:

```yaml
name: AI Code Review

on:
  pull_request:
    types: [opened, synchronize, reopened, ready_for_review]
  pull_request_review_comment:
    types: [created]

jobs:
  ai-review:
    if: github.event_name == 'pull_request' && github.event.pull_request.draft == false
    runs-on: ubuntu-latest
    permissions:
      pull-requests: write
      contents: read
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - run: git fetch origin ${{ github.base_ref }}
      - uses: actions/setup-node@v4
        with:
          node-version: "20"
      - run: npx -y @giolabsuy/ai-code-reviewer@latest review-pr
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}

  handle-feedback:
    if: github.event_name == 'pull_request_review_comment'
    runs-on: ubuntu-latest
    permissions:
      pull-requests: write
      contents: read
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "20"
      - run: npx -y @giolabsuy/ai-code-reviewer@latest handle-feedback
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          GITHUB_EVENT_PATH: ${{ github.event_path }}
          GITHUB_REPOSITORY: ${{ github.repository }}
          GITHUB_ACTOR: ${{ github.actor }}
```

import { Callout } from 'nextra/components'

<Callout type="info">
La variable `GITHUB_ACTOR` es requerida. El handler la usa para no responder a sus propios comentarios en un loop infinito.
</Callout>

## Cómo funciona

Cuando un desarrollador responde a un comentario inline del bot:

1. Se dispara el evento `pull_request_review_comment`
2. Arranca el job del workflow `handle-feedback`
3. El CLI lee el payload del evento desde `GITHUB_EVENT_PATH`
4. Verifica que la respuesta es a un comentario del bot (no de otro desarrollador)
5. Verifica que la respuesta empieza con `/explain` o `/dismiss`
6. Para `/explain`: envía el hallazgo original + el contexto del archivo al LLM y publica una respuesta detallada
7. Para `/dismiss`: resuelve el hilo del comentario y publica un breve acuse de recibo

## Auto-resolución en push

Cuando se suben nuevos commits a la rama del PR, el revisor verifica si alguna de las líneas marcadas anteriormente cambió. Si la línea de un hallazgo fue modificada en el nuevo push, el hilo del comentario se resuelve automáticamente y el bot responde confirmando que detectó el fix.

## Deshabilitar el dismiss

Para mantener los hallazgos visibles incluso después del reconocimiento del desarrollador (útil para auditoría), configurá:

```yaml
feedback:
  enabled: true
  allowDismiss: false
```

Con esta configuración, `/explain` sigue funcionando pero `/dismiss` es ignorado y el bot responde explicando que el dismiss está deshabilitado.
