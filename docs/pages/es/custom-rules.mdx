# Reglas personalizadas

Agregá reglas de revisión específicas del proyecto que la IA debe seguir. Tus reglas se agregan al template built-in del stack tecnológico y tienen precedencia en caso de conflictos.

## Configuración

1. Creá un archivo `code-review-rules.md` en la raíz de tu repo (o en cualquier ruta que prefieras).
2. Referencíalo en `.ai-review.yml`:

```yaml
rules: ./code-review-rules.md
```

O usá el campo inline `customInstructions:` para snippets cortos:

```yaml
customInstructions: |
  Este proyecto usa Clean Architecture. Cualquier import desde la capa
  de dominio a infraestructura es un hallazgo crítico.
```

## Formato de las reglas

Escribí tus reglas en Markdown plano. Usá headers para secciones, listas de bullets para reglas. Sé directo y específico — el modelo lee esto como parte de su system prompt.

```markdown
# Reglas del equipo

## Arquitectura

- Este proyecto sigue Clean Architecture. Capas: domain → application → infrastructure → presentation.
- Los archivos de domain (`src/domain/**`) no deben importar de ninguna otra capa.
- Los archivos de application (`src/application/**`) solo pueden importar de domain.
- Cualquier violación entre capas es un hallazgo `critical`.

## Manejo de errores

- Nunca usar `console.log` para errores. Usar la clase `Logger` de `src/shared/logger.ts`.
- Todos los métodos de servicio deben tener un `try/catch` que convierta errores desconocidos a `AppError`.

## Naming

- Las interfaces de repositorios deben terminar con `Repository` (ej. `UserRepository`).
- Las implementaciones de servicios deben terminar con `Service`.
- Los DTOs deben terminar con `Dto` (ej. `CreateUserDto`).
```

## Ejemplos del mundo real

### NestJS + Clean Architecture

```markdown
# Reglas del proyecto

## Enforcement de Clean Architecture

- `src/domain/**` — Sin imports externos. Solo lógica de negocio pura y tipos.
- `src/application/**` — Solo puede importar de `domain`. Sin decoradores de NestJS acá.
- `src/infrastructure/**` — Puede importar de `domain` y `application`. El glue de NestJS va acá.
- Marcar cualquier violación como `critical`.

## DTOs

- Todos los DTOs en `src/application/dto/` deben extender `BaseDto`.
- Campos requeridos usan `@IsNotEmpty()`. Campos opcionales usan `@IsOptional()`.
- Arrays usan `@IsArray()` + `@ArrayMinSize(1)`.

## Base de datos

- Los repositorios deben implementar la interface de `domain`. Nunca llamar TypeORM directamente desde un servicio.
- Todas las queries que pueden devolver múltiples filas deben usar `.take()` y `.skip()` para paginación.
```

### Reglas para Next.js

```markdown
# Reglas de equipo Next.js

## Componentes

- Los Server Components son el default. Agregá `use client` solo cuando uses hooks, event handlers o APIs del browser.
- Todos los componentes `use client` deben estar en `src/components/client/`. Documentar por qué se necesita la directiva.

## Data fetching

- Nunca hacer fetch de datos dentro de componentes cliente con `useEffect`. Mover el data fetching al Server Component padre.
- Las llamadas de mutación usan Server Actions, no API routes, a menos que la ruta deba ser llamada desde fuera de Next.js.

## Variables de entorno

- Los secrets solo en `.env.local`. Ningún secret puede tener el prefijo `NEXT_PUBLIC_`.
- Todas las env vars deben declararse en `.env.example` con un valor placeholder y un comentario.
```

## Tips

- **Sé específico con la severidad**: si querés que una regla bloquee PRs, escribí "Marcar como `critical`" o "Marcar como `major`". El modelo lo sigue.
- **Referenciá rutas reales**: "import de `src/domain/**`" es más claro que "import de la capa de dominio".
- **No repitas reglas built-in**: el template del stack ya cubre patrones comunes. Solo agregá lo específico de tu proyecto.
- **Mantenelas concisas**: las reglas muy largas tienen menos peso. Una oración específica vale más que un párrafo.
