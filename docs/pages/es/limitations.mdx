# Limitaciones

Restricciones y comportamientos conocidos a tener en cuenta antes de depender del revisor en producción.

## Limitaciones del LLM

### Falsos positivos y falsos negativos

El revisor usa un modelo de lenguaje. A veces va a marcar código que es correcto (falsos positivos) y va a perderse bugs que existen (falsos negativos). Tratá los hallazgos como señales para investigar, no como veredictos.

### Números de línea alucinados

El modelo puede citar un número de línea que no coincide exactamente con el diff. Siempre leé la descripción del hallazgo y mirá el código circundante, no solo la línea citada.

### Límites de ventana de contexto

Cada diff de archivo se trunca a 80,000 caracteres. Archivos muy grandes o PRs con muchos archivos modificados pueden tener algunos diffs truncados. El revisor registra qué archivos fueron truncados.

### Sin ejecución

El revisor lee código de forma estática. No puede ejecutar tests, correr el binario ni reproducir bugs en runtime. Los problemas dinámicos (condiciones de carrera, memory leaks, comportamiento bajo carga real) son más difíciles de detectar.

## Limitaciones de GitHub

### Solo las líneas visibles en el diff reciben comentarios inline

La API de GitHub solo permite comentarios inline en líneas que aparecen en el diff del PR. Los hallazgos en líneas de contexto no modificadas o archivos no incluidos en el diff se mueven al comentario resumen.

### Rate limits

Para PRs con muchos archivos, el revisor hace múltiples llamadas a la API para obtener el contenido de los archivos. Si se alcanza el rate limit de la API de GitHub, la revisión se reintenta con backoff.

### Los PRs en draft se omiten

El workflow recomendado omite los PRs en draft (`github.event.pull_request.draft == false`). Esto es intencional — los PRs en draft son trabajo en progreso.

## Contextos soportados

### `review-pr` requiere GitHub Actions

El comando `review-pr` lee el contexto del PR desde las variables de entorno `GITHUB_EVENT_PATH` y `GITHUB_REPOSITORY` que establece GitHub Actions. No puede ejecutarse fuera de un entorno de GitHub Actions.

### `handle-feedback` requiere Actions + trigger `pull_request_review_comment`

El comando `handle-feedback` requiere que el workflow sea disparado por `pull_request_review_comment`. Ver [Feedback inline](./handle-feedback).

### El grafo de dependencias es solo para JS/TS

El grafo de dependencias de un nivel (contexto de callers/callees) solo funciona para stacks JavaScript y TypeScript. Otros stacks envían solo los diffs de los archivos modificados.

## Limitaciones de configuración

### El override `tech:` es por repo, no por directorio

La detección del stack tecnológico aplica a todo el repo. Los monorepos con múltiples stacks obtienen las reglas del stack que sea detectado (o forzado vía `tech:`).

### El archivo `rules:` es un solo archivo

El campo `rules:` acepta un archivo Markdown. Si necesitás reglas de múltiples fuentes, concatínalas en un solo archivo, o usá `customInstructions:` para adiciones cortas.

## Costo

Cada revisión de PR es una llamada a la API del LLM. El costo depende del proveedor, el modelo y el tamaño del diff. Con `gpt-4o-mini` a los precios de OpenAI, un PR típico con 5–10 archivos modificados cuesta menos de $0.01. Los PRs más grandes con muchos archivos modificados cuestan proporcionalmente más.

Para limitar el costo, configurá `maxFileSize` para omitir archivos grandes generados o minificados, y usá patrones `ignore:` para excluir código no productivo.
