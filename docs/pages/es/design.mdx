# Diseño

Cómo funciona el revisor internamente — útil si estás debuggeando, extendiendo o simplemente querés entender las decisiones de diseño.

## Pipeline

Cada comando de revisión sigue el mismo pipeline:

```
CLI → resolución de config → detección de tech → ensamblado de prompt → llamada al LLM → output/publicación
```

1. **CLI** (`src/cli.ts`) — parsea el comando y flags vía Commander, delega al orquestador
2. **Resolución de config** (`src/config.ts`) — mergea `.ai-review.yml` sobre `DEFAULT_CONFIG`, carga el template built-in del stack desde `templates/`, carga el markdown de reglas del usuario
3. **Detección de tech** (`src/tech-detect.ts`) — detecta el stack desde deps de `package.json` o archivos indicadores; los stacks más específicos se verifican primero (NestJS antes que Node, Next.js antes que React)
4. **Ensamblado de prompt** (`src/prompts.ts`) — construye el system prompt (rol + reglas de tech + categorías de checks + escala de severidad + idioma) y el user prompt (metadata del PR + diffs de archivos, truncados a 80k chars)
5. **Llamada al LLM** (`src/openai.ts`) — envía el request con `response_format: json_schema`; el modelo está restringido al shape del `REVIEW_SCHEMA`
6. **Output/publicación** — filtra por severidad, ordena por nivel; para `review-pr`, publica comentarios inline vía Octokit + comentario resumen

## Output estructurado

El revisor usa `response_format: json_schema` de OpenAI para restringir la respuesta del modelo a una forma precisa. No hay regex ni parseo de string del output del LLM — el modelo devuelve JSON válido que coincide con el schema, o la API lo rechaza.

El schema garantiza:
- `findings[]` — cada uno con `file`, `line`, `severity`, `category`, `title`, `description`, `suggestion`
- `recommendation` — `approve` | `comment` | `request_changes`
- `overallScore` — número 0–10
- `summary` — string
- `anticipatedBugs[]` — lista de strings
- `regressionRisks[]` — lista de strings

## Nunca auto-aprueba (por defecto)

El mapeo de `recommendation` del LLM al evento de revisión de GitHub hace downgrade explícito de `approve` → `COMMENT`:

```typescript
private mapRecommendationToEvent(recommendation: Recommendation): ReviewEvent {
  if (recommendation === 'request_changes') return 'REQUEST_CHANGES';
  return 'COMMENT'; // approve → COMMENT; solo los revisores humanos aprueban
}
```

El auto-approve está disponible como feature opt-in. Ver [Auto-aprobar](./auto-approve).

## Grafo de dependencias (solo JS/TS)

Para stacks JavaScript y TypeScript, el revisor construye un grafo de dependencias de un nivel de los archivos modificados antes de enviar al LLM:

1. Parsea las sentencias `import` de cada archivo modificado
2. Resuelve las rutas importadas relativas al archivo
3. Lee el contenido de esos archivos importados (un nivel de profundidad)
4. Incluye ese contexto en el user prompt junto a los diffs

Esto le da al modelo visibilidad sobre los callers y dependencias, habilitando la detección de riesgos de regresión más allá de los archivos modificados.

## Placement de comentarios inline

Al publicar comentarios de revisión del PR, `buildDiffLineMap()` en `src/github.ts` parsea el diff unificado para determinar qué posiciones de línea en el diff son comentables. Solo las líneas que aparecen en el diff (adiciones, líneas de contexto) pueden recibir comentarios inline; la API de GitHub rechaza comentarios en líneas fuera del diff.

Los hallazgos que no se pueden mapear a una línea del diff se recolectan como "huérfanos" y se agregan al cuerpo del comentario resumen en lugar de descartarse.

## Código de salida

`review-pr` termina con código `1` cuando la recomendación es `request_changes`. Esto permite que las reglas de branch protection bloqueen merges cuando el revisor marca problemas graves, sin necesidad de configuración adicional en el workflow.

## Decisiones de diseño clave

| Decisión | Fundamento |
|---|---|
| Output estructurado sobre parseo | El output del LLM está restringido por el schema de la API — sin regex frágiles ni prompt engineering para el formato de output |
| Nunca auto-aprueba por defecto | La aprobación humana es una puerta deliberada; el bot no debería eliminarla sin opt-in |
| Reglas personalizadas sobreescriben built-in | Las reglas del usuario se agregan al final del system prompt, dándoles la mayor prioridad |
| Código de salida 1 en request_changes | La integración con CI no requiere configuración extra — las reglas de branch protection simplemente funcionan |
| Truncado a 80k chars por archivo | Mantiene el tamaño del prompt razonable mientras cubre la mayoría de los diffs realistas |

## Archivos fuente

| Archivo | Responsabilidad |
|---|---|
| `src/cli.ts` | Entry point CLI con Commander; definición de comandos; template `EXAMPLE_CONFIG` |
| `src/reviewer.ts` | Orquestador; una función por comando; `parseLocalDiff()` |
| `src/config.ts` | Carga de config; glob-matching para patrones de ignore; cargador de template built-in |
| `src/tech-detect.ts` | Detección de stack; ordenada de específico a genérico |
| `src/prompts.ts` | Ensamblado de prompts system y user; truncado de diffs |
| `src/openai.ts` | Llamada al LLM; `REVIEW_SCHEMA`; `ReviewResult` tipado |
| `src/github.ts` | Integración con Octokit; `buildDiffLineMap()`; publicación de comentarios inline |
| `src/output.ts` | Output en terminal; reporte Markdown; filtrado y ordenamiento por severidad |
| `src/types.ts` | Todos los tipos compartidos: `Severity`, `CheckCategory`, `TechStack`, `ReviewerConfig`, etc. |
